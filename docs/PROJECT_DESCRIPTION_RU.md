# Описание проекта решения: РМК «Обходчик» (Field Inspector)

**Кратко:** экосистема для организации обходов промышленного оборудования: веб-панель руководителя, REST API, общая облачная БД (Supabase) и мобильное приложение для полевых сотрудников. Документ отражает текущую реализацию в репозитории `field_inspector_web` и согласован с кодом и `README.md` в корне проекта.

---

## 1. Название проекта и краткое описание решения

**Название:** РМК «Обходчик» / Field Inspector.

**Суть:** цифровая поддержка инспекционных обходов на промышленных площадках: иерархия объектов (завод — площадка — цех — оборудование), назначение заданий обходчикам, фиксация отчётов и дефектов в поле, контроль входящих заявок от работников и аналитика для руководства. Решение сочетает веб-интерфейс администратора, серверный API для демо-маршрутов и чек-листов и единое хранилище операционных данных в Supabase (PostgreSQL), с которым работает и мобильный клиент (Flutter).

---

## 2. Проблема и контекст

**Проблема:** при ручном или разрозненном учёте обходов сложно гарантировать полноту маршрута, прозрачность статусов, своевременную передачу дефектов и единую картину по смене для диспетчера.

**Контекст:** типичный сценарий — крупная площадка с иерархией оборудования; руководитель формирует задания с перечнем точек обхода; работник выполняет их в поле с мобильного устройства; администратор видит заявки, задания, отчёты и дефекты в панели. До внедрения единой системы данные часто дублируются в таблицах и мессенджерах без связи с нормативной иерархией объектов.

---

## 3. Целевая аудитория и основной потребитель

| Сегмент | Роль | Потребность |
|--------|------|-------------|
| **Диспетчер / руководитель смены** | Веб-панель | Обзор смены, заданий, обходчиков, дефектов, аналитики; создание заданий и учётных записей обходчиков; рассмотрение заявок на осмотр. |
| **Администратор ИТ / внедрение** | Развёртывание | Настройка Supabase, переменных окружения, CORS, согласование режимов входа (Supabase vs API). |
| **Обходчик (полевой сотрудник)** | Мобильное приложение | Список назначенных заданий, прохождение маршрута, чек-листы, отчёты, заявки на дополнительный осмотр. |

Основной потребитель продукта в операционном контуре — **диспетчер и обходчик**; **ИТ** — потребитель документации и инфраструктуры.

---

## 4. Идея и логика предлагаемого решения

**Идея:** разделить **операционные данные** (профили, задания, назначения, отчёты, заявки), хранимые в **Supabase** и доступные веб-клиенту и мобильному приложению, и **демонстрационный слой API** на **FastAPI + SQLite** (маршруты обходов, оборудование, конструктор шаблонов чек-листов) для разработки и демо без обязательной связки с полевым контуром.

**Логика:** администратор авторизуется (предпочтительно через Supabase Auth); работает с деревом объектов `equipment_nodes`, создаёт задания с пунктами `inspection_task_items` и назначениями `inspection_task_assignments`; обходчик синхронизируется с тем же проектом Supabase; дашборд агрегирует метрики из этих таблиц. Веб-панель не обязана дублировать весь функционал мобильного клиента, но даёт сквозной контроль и отчётность.

---

## 5. Анализ аналогов и отличий продукта

| Класс аналогов | Отличие «Обходчика» |
|----------------|---------------------|
| Универсальные CMMS / EAM | Узкая специализация под **обход по маршруту** и **чек-листы по оборудованию**, а не полный цикл MRO. |
| Простые таск-трекеры | Есть **иерархия промышленных объектов**, привязка заданий к **конкретным единицам оборудования**, отчёты с признаками дефекта. |
| Только web или только mobile | **Явное разделение** веб-админки и мобильного исполнителя с общей БД Supabase. |

Риск: при неполной интеграции режимов авторизации (см. раздел 9) конкурентное преимущество «единого окна» снижается — это зафиксировано как ограничение текущей версии.

## 6. Сценарии использования

1. **Назначение обхода с площадки:** администратор открывает раздел «Объекты», выбирает оборудование чекбоксами, создаёт задание с маршрутом для мобильного приложения.
2. **Просмотр заданий:** список заданий, карточка задания, переход к связанным отчётам.
3. **Входящие заявки:** работник отправил заявку на осмотр (`inspection_task_requests`); администратор просматривает список и меняет статус (одобрено / отклонено).
4. **Создание учётной записи обходчика:** через FastAPI с ролью service role Supabase (при настроенном бэкенде и ключах).
5. **Контроль смены:** дашборд показывает агрегаты по заданиям, назначениям, отчётам и дефектам за день (данные из Supabase).
6. **Аналитика и дефекты:** сводные цифры и списки дефектов по отчётам `inspection_reports`.
7. **Демо-маршрут API:** при работе только с FastAPI — прохождение демонстрационного маршрута и чек-листа без Supabase.

## 7. Описание продукта

**Продукт** — монорепозиторий:

| Компонент | Назначение |
|-----------|------------|
| **Веб-приложение** (`frontend/`) | SPA на Vite + React + TypeScript + MUI: дашборд, объекты, задания, заявки, обходчики, топология, отчёты, аналитика, дефекты, настройки, конструктор чек-листов (часть сценариев завязана на Supabase). |
| **API** (`backend/`) | FastAPI, асинхронный SQLAlchemy, SQLite по умолчанию: `/api/v1/auth`, дашборд-снимок (legacy-таблицы), маршруты, оборудование, шаблоны чек-листов, админ-операции с Supabase при наличии ключей. |
| **Supabase** | Postgres + Auth + RLS; таблицы `profiles`, `inspection_tasks`, `inspection_task_items`, `inspection_task_assignments`, `inspection_task_requests`, `inspection_reports`, `equipment_nodes` и др. (см. `supabase/migrations/`). |
| **Мобильное приложение** (`mobile/`) | Flutter: задания, осмотр, отчёты, заявки; тот же проект Supabase. |

**Маршрутизация веб-приложения (основные пути):** `/` (дашборд), `/workers`, `/objects`, `/topology`, `/tasks`, `/tasks/create`, `/tasks/:id`, `/task-requests`, `/planning`, `/checklist-builder`, `/defects`, `/analytics`, `/reports`, `/route/:id` и др. (см. `frontend/src/app/router/index.tsx`).

## 8. Функционал продукта

**Веб-панель (реализовано в коде):**

- Авторизация: Supabase и/или JWT FastAPI (режимы описаны в README).
- **Дашборд:** KPI, сотрудники, события, прогресс по объектам, критический дефект — из агрегации Supabase (`dashboardAdminApi`).
- **Объекты:** дерево `equipment_nodes`, выбор оборудования, создание патрульного задания.
- **Задания:** список, создание/детализация, связь с пунктами и назначениями.
- **Входящие заявки:** таблица `inspection_task_requests`, действия одобрения/отклонения.
- **Обходчики:** список профилей, создание учётной записи через API (при доступности).
- **Топология / схема объектов:** работа с узлами и связями.
- **Дефекты, аналитика, отчёты:** выборки из `inspection_reports` и связанных сущностей.
- **Конструктор шаблонов чек-листов:** через API FastAPI.
- **Планирование, настройки:** страницы в макете (часть — заглушки).

**Мобильное приложение:** список заданий, детали, осмотр, отчёты, заявка на задачу, локализация (см. `mobile/lib/`).

**API (FastAPI):** health, JWT-login, dashboard bundle (SQLite), routes, equipment, checklist templates, sync, admin workers и др. (см. `backend/app/api/`).

## 9. Архитектура и IT-инфраструктура

**Клиент:** браузер → статическая сборка SPA или dev-сервер Vite; в dev запросы к `/api` и `/health` могут проксироваться на локальный FastAPI.

**Сервер приложений:** процесс Uvicorn с FastAPI; опционально за reverse-proxy в продакшене.

**Данные:**

- **SQLite** — файл рядом с процессом API (или путь из `DATABASE_URL`); не совпадает с Supabase.
- **Supabase** — управляемый Postgres, Auth, Storage по настройкам проекта; RLS на таблицах согласно миграциям.

**Клиент Supabase в браузере** использует publishable-ключ; **service role** допускается только на сервере (бэкенд `.env`).

**Схема потоков (упрощённо):**

```
Браузер (React) ──► Supabase (Postgres + Auth)     Мобильное приложение ──► Supabase
       │
       └──► FastAPI + SQLite  (JWT, демо-маршруты, чек-листы)
```

**Развёртывание:** сборка `frontend/dist`, хостинг статики; отдельно сервис API; проект Supabase в облаке; секреты через переменные окружения, не в git (`backend/.env`, `frontend/.env.local` в ignore).

## 10. Принятые технические решения и компромиссы

1. **Два источника данных (Supabase и SQLite API):** позволяет развивать демо и продакшен-данные параллельно; компромисс — необходимость явно выбирать режим разработки и не смешивать вход Supabase с вызовами API без «моста» (см. README, раздел 7).
2. **Прямые запросы PostgREST из веб-клиента** к Supabase для операционных сущностей — скорость разработки и RLS на стороне БД; компромисс — логика размазана между клиентом и политиками SQL.
3. **Дашборд из агрегации на клиенте** — без отдельного OLAP-сервиса; компромисс — при больших объёмах может понадобиться материализованные представления или API-слой.
4. **Нормализация отображения** (номер смены, подписи площадки): эвристики поверх «грязных» тестовых данных; не заменяет дисциплину ввода в НСИ.
5. **Одобрение заявок:** обновление строки в Postgres; автоматическое создание задания из заявки может быть расширено отдельной задачей.

## 11. Требования к решению

**Функциональные (из текущей реализации):**

- Роли admin/worker в `profiles`; доступ к строкам по RLS в Supabase.
- Создание и ведение заданий с пунктами оборудования и назначениями на работников.
- Приём отчётов и фиксация дефектов из поля (мобильное + таблицы отчётов).
- Просмотр заявок и смена статуса заявки администратором.

**Нефункциональные:**

- Разделение секретов: publishable в клиенте, service role только на сервере.
- CORS для API: перечень origin должен включать реальный адрес Vite (в т.ч. порт 5174 при необходимости).
- Сборка фронтенда без утечки ключей в публичные репозитории.

**Соответствие MVP:** перечисленное покрывает сценарий «назначили обход — выполнили — зафиксировали — увидели на панели» при настроенном Supabase и активных пользователях.

## 13. Риски и ограничения

1. **Гибридный вход:** одновременное включение Supabase и `VITE_API_BASE_URL` без обмена токенов ведёт к 401 на API — риск ошибок конфигурации.
2. **Рассинхрон схемы:** миграции в репозитории и фактическая БД должны совпадать с полями, которые пишет клиент.
3. **Данные дашборда** зависят от качества полей `shift_label`, `site_name`, `area_name` в заданиях; мусорные значения маскируются эвристиками, но не заменяют НСИ.
4. **Масштабирование:** агрегаты на клиенте без пагинации на всех экранах могут стать узким местом при росте данных.
5. **Юридические и организационные:** персональные данные работников — в рамках политики компании и настроек Supabase (регион, DPA).
