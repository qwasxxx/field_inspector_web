# Архитектура: «Мобильный офис» и роль веб-приложения

Документ согласован со схемой **«Мобильный офис»** и сценарием **«Руководитель производственного звена ↔ Обходчик»**.

## Роли

| Роль | Клиент | Задачи |
|------|--------|--------|
| **Обходчик** | Мобильное приложение | Выполнение задания по обходу → чек-лист → фото/видео → отправка результатов |
| **Руководитель производственного звена (админ)** | **Это веб-приложение** (`admin_panel`) | Планирование: создание обходов, назначение на смену/день; подготовка шаблонов чек-листов; контроль и анализ поступающих данных |

Полевой обходчик и руководитель не дублируют один и тот же интерфейс: в целевой системе данные сходятся в **сервисах мобильного офиса** и **единой БД**.

## Целевая интеграция (не реализована в MVP)

```text
[Обходчик] ──► Мобильное приложение ──┐
                                       ├──► Сервисы мобильного офиса ──► БД
[Руководитель ПЗ] ──► Веб-приложение ──┘              │
                                                      └──► Внешние системы: ТОиР/ТОРО, НСИ, промышленные ИС, служба аутентификации
```

В текущем MVP вместо API используются **localStorage** и **имитация очереди синхронизации** — чтобы отработать UX и доменную модель до подключения бэкенда.

## Соответствие экранов сценарию руководителя

1. **Планирование обходов** (`/planning`) — заглушка под будущее назначение заданий на смену/день через сервисы мобильного офиса.
2. **Конструктор чек-листов** (`/checklist-builder`) — ручное создание шаблонов для последующего использования обходчиками (после интеграции — выгрузка в НСИ/сервисы).
3. **Обзор** (`/`) — контроль статусов маршрутов и очереди «отправки» (в продуктиве — аналитика результатов обходов).
4. **Просмотр маршрута** (`/route/:id`) — в продуктиве основной сценарий обхода на **мобильном клиенте**; в веб-MVP экран допускается как **демо-проверка** маршрутов и шаблонов.

## Кодовая структура

Feature-based слоение (`entities/`, `features/`, `pages/`, `widgets/`) сохраняется для последующей подстановки **API-клиентов** вместо mock и localStorage без смены сценариев UI.
